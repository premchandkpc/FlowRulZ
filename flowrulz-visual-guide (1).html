01 — end to end

Message lifecycle

The same five stages handle every mode. Only the final hop and what happens inside the VM box differ.

02 — worked example

Service A → Gate → Service B or Service C

One concrete rule: service A publishes an order. A single Gate on amount decides which downstream service handles it — no branching logic lives in A, B, or C, only in the compiled plan.

schema:{!order_id:string,!amount:float} g:amount>10000 n:service-b f:service-c

Why f: needs no extra wiring

Fallback is positional in the DSL — f:service-c directly follows the Gate, so a false comparison jumps straight to it. Nothing in service A, B, or C's code changes if the threshold moves.

Same shape, more branches

This is the two-way case. Swap the single Gate for a dag:{...} or chain more Gates to route across three, four, or N services on the same event.

03 — dsl → bytecode

Compilation pipeline

A one-line DSL string becomes a typed, scored, bincode-serializable plan. Type checking runs as a pre-pass before any instruction is emitted.

schema:{!amount:float} g:amount>10000 n:manual-review r3:exp f:auto-approve

Instruction encoding

Every opcode packs into one 8-byte struct — opcode·flags·a·b·c — so a plan is just a flat, cache-friendly array.

Complexity scoring

Next/Async score 10, Parallel/DAG score 20, Chunk 25 — summed at compile time to route the plan onto the fast, normal, or heavy lane.

04 — execution

VM dispatch loop

A stackless interpreter walks the instruction array with an instruction pointer, switching on opcode. Services enrich the ExecutionContext rather than replacing it.

05 — membership

Single-leader cluster

No Raft, no Paxos. Nodes sort by ID; the lowest alive node is leader. Gossip converges membership faster than heartbeats alone, and a lease-based check triggers re-election within one lease window of a leader going dark.

t=0 — node-a is leader, term 3. Gossip pushing membership state to 2 random peers every 2s.

06 — rollout

Plan distribution & quorum

The leader compiles, publishes, waits for a majority of followers to ACK, then activates. Old versions keep serving in-flight executions until the switch.

Leader compiles order-flow v17 and publishes to _flowrulz_plans.

07 — deployment

Kubernetes topology

FlowRulZ nodes need stable network identity for gossip and seed discovery, so they run as a StatefulSet, not a Deployment. A headless service gives every pod a DNS-resolvable address for the Cluster Bus; a regular ClusterIP service load-balances client traffic across whichever pod picks it up.

leader pod active request path ┄ dashed = cluster bus / gossip / reroute

Why StatefulSet, not Deployment

Leader election sorts nodes by NodeID — that has to be stable across pod restarts, or the lowest-ID rule reshuffles leadership on every rollout. Ordinal pod names (flowrulz-0, flowrulz-1…) give that for free.

Two services, two jobs

The headless service (ClusterIP: None) resolves directly to pod IPs for gossip and _flowrulz_plans traffic. The regular ClusterIP service is what producers and consumers actually connect to — any pod can accept the connection.

08 — declarative routing

DAG execution

dag:{A:[],B:[A],C:[A],D:[B,C]} compiles into layers. Each layer runs in parallel; parent results deep-merge into every dependent's input before it fires.

09 — allocation

Memory model

Arena bump allocator

Per-message temporaries bump-allocate at O(1); the whole arena resets between messages instead of tracking individual frees.

bump pointer: 0 bytes

String interner

Lock-free, deduplicated. Common strings (HTTP headers, service names) are prefilled at startup and reused by ID.

interning "content-type" → id 3

10 — go ↔ rust

FFI callback bridge

Rust owns the hot path; Go owns transport and admin. A synchronous call crosses the boundary three times per service hop.
